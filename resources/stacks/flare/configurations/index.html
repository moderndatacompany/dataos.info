    Attributes of Flare Configuration¶

    Flare Stack can be configured using attributes in a declarative YAML format.

    Structure of Flare Stack-specific Section¶

    flare:
  driver:
    coreLimit: {{1000m}} 
    cores: {{1}}
    memory: {{1024m}}
  executor: 
    coreLimit: {{1000m}}
    cores: {{1}}
    instances: {{1}}
    memory: {{1024m}}
  job:
    explain: {{true}}
    logLevel: {{INFO}}
    streaming: # Streaming Section
      {}
    inputs: #Inputs Section
      {}
    outputs: #Outputs Section
      {}
    steps: # Steps Section
      {}
    assertions: # Assertions Section
      {}
    actions: # Actions Section
      {}

    The below table summarizes the various attributes within the Flare Stack-specific Section:

    Attributes Data Type Default Value Possible Value Requirement
    flare object none none mandatory
    driver object none none optional
    executor object none none optional
    job object none none mandatory
    explain boolean false true/false optional
    logLevel string INFO INFO / WARN /
    DEBUG/ ERROR    		 optional
    streaming object none none optional
    inputs object none none mandatory
    steps object none none optional
    outputs object none none mandatory
    assertions object none none optional
    actions object none none optional

    Refer to the link here for more information regarding the Flare Stack-specific Section attributes, click here.

    A Flare Job can be further split into 5 sections:

    Inputs¶

    The inputs section encompasses crucial configuration settings necessary for extracting data from diverse sources, such as name, format, dataset address, etc. To facilitate reading data from multiple sources, an array of data source definitions are required. To obtain more detailed information, kindly refer to the link below.

    Flare Input Section Grammar¶

    Within the Flare Job's inputs section, users can define the configuration details for input datasets. This allows the Flare Job to read data from different supported data sources. It is worth mentioning that a single Flare Job can handle multiple data sources simultaneously.

    Structure of the inputs Section¶

    Common Configuration¶

    inputs:
  - name: {{account_connect}}
    dataset: {{dataos://gcdexport:none/gcdcore_account}}
    format: {{csv}}
    Flare Input Section Common YAML Configuration

    Advanced Configuration¶

    inputs:
  - name: {{account_connect}}
    dataset: {{dataos://gcdexport:none/gcdcore_account}}
    query: {{select * from icebase.retail.city limit 100}}
    format: {{csv}}
    isStream: {{true}}
    schemaType: {{avro}}
    schemaPath: {{dataos://thirdparty:none/somedir/someschema.avsc}}
    schemaString: {{"{avsc_schema_file_content}"}}
    schemaSubject: {{custom-topic-value-schema-name}}
    schemaId: {{2}}
    options:
      {{key1: value1}}
      {{key2: value2}}
    incremental:
      context: {{incrinput}}
      sql: {{select ws_sold_date_sk, ws_sold_time_sk, ws_item_sk, ws_bill_customer_sk, ws_web_page_sk, ws_ship_mode_sk, ws_order_number, ws_quantity, ws_list_price,ws_sales_price, ws_wholesale_cost, ws_net_profit from incrinput where ws_sold_date_sk between '$|start_date|' AND '$|end_date|'}}
      keys:
        - name: {{start_date}}
          sql: {{select 2452641}}
        - name: {{end_date}}
          sql: {{select 2452642}}
      state:
        - key: {{start_date}}
          value: {{end_date}}

    Flare Input Section Advanced YAML Configuration

    Configuration Fields¶

    inputs¶

    Description: All input datasets and their specific properties are declared within this section
    Data Type: Object
    Requirement: Mandatory
    Default Value: None
    Possible Value: None
    Example Usage:

    inputs:
  {}

    name¶

    Description: This field represents the assigned name to the input. It serves as a reference for the input and can be used for querying the input using Spark SQL. Consider this field as a view in Spark, allowing you to interact with the input data.
    Data Type: String
    Requirement: Mandatory
    Default Value: None
    Possible Value: None
    Example Usage:

    name: account_connect

    dataset¶

    Description: Represents the UDL address of the dataset from which you want to read data. You can also load a specific file within the dataset. For example, dataos://gcdexport:none/gcdcore_account/account_x.csv.
    Data Type: String
    Requirement: Mandatory
    Default Value: None
    Possible Value: Any valid UDL address
    Example Usage:

    dataset: dataos://gcdexport:none/gcdcore_account

    query¶

    Description: Flare supports reading from Minerva. The field allows you to specify a Minerva SQL directly. This enables you to load the result of the specified query and make it available as a view in the Spark session.
    Data Type: String
    Requirement: Mandatory
    Default Value: None
    Possible Value: Any valid Minerva SQL query
    Example Usage:

    query: select * from icebase.retail.city limit 100

    format¶

    Description: Format of the dataset
    Data Type: String
    Requirement: Optional
    Default Value: iceberg
    Possible Value: iceberg/text/json/parquet/orc/avro/csv/hudi/xml/db/xlsx
    Example Usage:

    format: csv

    isStream¶

    Description: The isStream field is used to determine whether to read a dataset in a streaming fashion. Set this flag to true if you want to read the dataset as a stream.
    Data Type: Boolean
    Requirement: Optional
    Default Value: If not supplied, the default value will be determined based on the depot type. For depots like Kafka and EventHub, the default value is true, while for depots like GCS and ABFSS, the default value is false.
    Possible Value: true/false
    Example Usage:

    isStream: true

    schemaType¶

    Description: The schemaType field specifies the type of schema provided in the schemaPath or schemaString field above.
    Data Type: String
    Requirement: Optional
    Default Value: avro
    Possible Value: avro/spark
    Example Usage:

    schemaType: avro

    schemaPath¶

    Description: Represents the DataOS address to the schema that will be applied when loading the dataset. This field is particularly useful for datasets with formats like CSV.
    Data Type: String
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    schemaPath: dataos://thirdparty:none/somedir/someschema.avsc

    schemaString¶

    Description: The schemaString field represents the Spark struct schema JSON string or Avro schema JSON.
    Data Type: String
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    schemaString: "{avsc_schema_file_content}"

    schemaSubject¶

    Description: The schemaSubject field represents the subject name of the schema associated with the input data. It can be used to override a subject name to refer to the schema of a Kafka topic in the schema registry when loading Avro data from it.
    Data Type: String
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    schemaSubject: custom-topic-value-schema-name

    schemaId¶

    Description: The schemaId field represents the ID of the schema associated with the input data. It is used when utilizing a schema with a specified schema ID in the Kafka schema registry.
    Data Type: Integer
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    schemaId: 2

    options¶

    Description: The options field allows you to specify additional options or configurations for the input. These options are provided as key-value pairs. You can use any other options supported by the underlying Spark connector to load data from the supplied dataset. Flare will iterate over these options and forward each one of them to Spark.
    Data Type: Map (String, String)
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    options:
  key1: value1
  key2: value2

    incremental¶

    Description: The incremental field allows you to load data from a dataset in an incremental fashion, you can use this configuration. This feature is particularly useful when dealing with large datasets and performing incremental updates.
    Data Type: Object
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    incremental:
  context: incrinput
  sql: select ws_sold_date_sk , ws_sold_time_sk , ws_item_sk, ws_bill_customer_sk , ws_web_page_sk ,ws_ship_mode_sk, ws_order_number, ws_quantity,ws_list_price ,ws_sales_price , ws_wholesale_cost  , ws_net_profit from incrinput where ws_sold_date_sk between '$|start_date|' AND '$|end_date|'
  keys:
    - name: start_date
      sql: select 2452641
    - name: end_date
      sql: select 2452642
  state:
    - key: start_date
      value: end_date

    context¶

    Description: The context or name of the incremental input.
    Data Type: String
    Requirement: Mandatory for incremental processing
    Default Value: None
    Possible Value: Any string
    Example Usage:

    context: incrinput

    sql¶

    Description: The SQL query used to retrieve the incremental data from the input. It should specify the columns and conditions for the incremental processing.
    Data Type: String
    Requirement: Mandatory
    Default Value: None
    Possible Value: None
    Example Usage:

    sql: select ws_sold_date_sk, ws_sold_time_sk, ws_item_sk, ws_bill_customer_sk, ws_web_page_sk, ws_ship_mode_sk, ws_order_number, ws_quantity, ws_list_price,ws_sales_price, ws_wholesale_cost, ws_net_profit from incrinput where ws_sold_date_sk between '$|start_date|' AND '$|end_date|'

    keys¶

    Description: The keys used for incremental processing. Each key contains a name and an SQL query that retrieves the corresponding value for the key.
    Data Type: List of Objects
    Requirement: Mandatory for incremental processing
    Default Value: None
    Possible Value: None
    Example Usage:

    keys:
  - name: start_date
    sql: select 2452641
  - name: end_date
    sql: select 2452642

    state¶

    Description: The state or key-value pairs used to store the incremental processing state. Each state entry consists of a key and its corresponding value.
    Data Type: List of Objects
    Requirement: Optional
    Default Value: None
    Possible Value: None
    Example Usage:

    state:
  - key: start_date
    value: end_date

    Steps¶

    The steps section outlines the prescribed sequence of tasks for data enrichment and processing, including but not limited to the addition of calculated columns, aggregations, and filtering via SQL. These are defined in sequence under steps. For further information, please consult the linked page.

    The streaming section encompasses properties that pertain to the execution of stream jobs. For further details, please refer to the page provided below.

    Assertions¶

    In the context of data quality, assertions refer to validation rules that are tailored to a particular business domain and serve to determine the fitness-for-purpose of datasets. The application of assertions enables the execution of additional validation checks on top of existing datasets, thereby enhancing their overall quality. Assertions are defined within the dedicated assertions section. To gain further insights, please refer to the following page.

    Assertions
